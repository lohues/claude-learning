MODULE 02

Trust & Permissions

● core ~22 min read 5 modes + rules 5 lessons · 1 check

Claude Code is read-only until you say otherwise. This module is the difference between supervised and unsupervised work — the modes, the live cycle, the rule system, and where settings actually live.

2.1 · Read-only by default, for a reason

The power that makes the loop useful — editing files, running bash — is the same power that makes it dangerous unsupervised.

Out of the box, Claude Code reads and explores freely but asks before it changes anything. Every edit, every shell command surfaces a prompt. That’s deliberate friction: it keeps a misread instruction from becoming a deleted directory. The whole skill of this module is loosening that friction exactly as much as the situation earns — no more.

Frame

Think of it as trust you grant in increments. A fresh clone of someone else’s repo earns very little. Your own project with 90% test coverage earns a lot. The modes below are the increments.

2.2 · The five modes (and a gated sixth)

Set with defaultMode in settings, or switched live. Use the console up top as you read — each name maps to a row of behavior.

  • default — reads auto-approve; edits and most bash prompt you. The safe starting point.
  • acceptEdits — auto-approves file edits and a specific set of filesystem bash commands on in-scope paths. Fast for your own repo. Not “approve everything.”
  • plan — research only. Reads, explores, writes you a plan, makes zero changes. The mode for “tell me what you’d do” before committing.
  • dontAsk — denies everything unless an allow rule matches. For locked-down headless/CI runs where only a known set of actions is permitted.
  • bypassPermissions — skips all checks (--dangerously-skip-permissions, the “YOLO” flag). Ephemeral, network-isolated sandboxes only.
Gotcha · the gated sixth: auto

A sixth mode, auto, only appears for qualifying accounts — it’s plan-, model-, and API-gated (broadly: Team/Enterprise/API plans on direct Anthropic API with a recent model, admin-enabled). The exact tier matrix shifts between releases, so verify against the live docs rather than trusting any blog’s list. If you don’t see it, that’s why.

2.3 · Shift+Tab cycles three — not all of them

The most common misconception in this whole module.

Pressing Shift+Tab mid-session rotates through exactly three modes:

live mode cycle
# press Shift+Tab to rotate:
default  →  acceptEdits  →  plan  →  (back to default)

# NOT in the cycle — set deliberately, not by accident:
dontAsk            settings.json or --permission-mode
bypassPermissions  --dangerously-skip-permissions flag
auto               appears only if your account qualifies

The design intent is sane: the two genuinely dangerous modes can’t be reached by fat-fingering a key combo. You have to choose them, in writing.

2.4 · Rules beat modes: allow / deny / ask

Modes are the blunt instrument. Rules are the scalpel — and they override the mode.

In settings.json, a permissions block lets you allow, deny, or force-ask specific tools and patterns. Edit it directly or via the /permissions command:

.claude/settings.json
{
  "permissions": {
    "allow": ["Bash(npm run test:*)", "Read(src/**)"],
    "deny":  ["Read(.env*)", "Bash(git push:*)"],
    "ask":   ["Bash(rm:*)"]
  }
}

The evaluation order on every single tool call:

decision order
PreToolUse hooks → deny rules → allow rules → ask rules → mode default
The load-bearing fact

Deny rules win — even in bypassPermissions. A deny: ["Read(.env*)"] holds no matter how reckless the mode. This is how you make even YOLO mode safe-ish: deny the handful of things that must never happen, then stop worrying about the rest.

2.5 · Where settings live (the hierarchy)

Same setting, four possible homes. Higher wins. Put each rule where its blast radius belongs.

1managed / enterpriseOrg admin policy. Overrides everyone — e.g. disabling bypass fleet-wide.
2.claude/settings.local.jsonYour personal, gitignored overrides for this repo. Not shared.
3.claude/settings.jsonProject settings, committed. The team’s shared rules.
4~/.claude/settings.jsonYour global defaults across every project.
  • Shared team guardrails (deny .env, force-ask on rm) → committed project settings, so everyone inherits them.
  • Personal speed tweaks → local settings, so you don’t impose them on teammates.
  • Admins lock down bypass entirely with disableBypassPermissionsMode at the managed level.
Gotcha · this is not a security sandbox

Permissions govern Claude’s tool calls; they are not an OS-level jail. A prompt injection hiding in a file Claude reads could still attempt actions. For untrusted code or content, pair permission rules with real sandboxing (containers, isolated VMs). Anthropic is explicit that permissions complement OS sandboxing — they don’t replace it.

Try it yourself

Tick as you go — these feed your meter.

  • Cycle Shift+Tab and watch the mode label changeconfirm it’s only default / acceptEdits / plan
  • Run /permissions and read your current allow/deny listsmost repos start empty
  • Add deny: ["Read(.env*)"] to project settingsyour first real guardrail
  • Ask Claude to read .env and watch the deny holdeven try it after switching to acceptEdits

You switch to acceptEdits to move faster on your own repo. Claude runs rm -rf dist/ without asking you first. Bug, or working as intended?
