Module 07 · three things people call “agents”

Running more than one —
without conflating three things.

“Use agents” could mean any of three completely different mechanisms with different shapes, costs, and maturity. Subagents, agent teams, and git worktrees are not synonyms. Pick the wrong one and you’ll pay 15× the tokens for coordination you didn’t need — or fight file collisions you could have avoided.

The three topologies — select one to see its shape and trade-offs
MODULE 07

Going Parallel

● core ~28 min read teams are experimental 6 lessons · 1 check

The three mechanisms kept strictly apart: subagents (isolation in one session), agent teams (coordinating sessions, experimental), git worktrees (filesystem isolation) — then the framework for choosing, and the honest case for staying single.

7.1 · Three answers, three shapes

The word “agents” is doing too much work. Separate it into three.

  • Subagents — a star. One session, a lead delegating to isolated workers that report back one-way. For keeping work out of your main context window.
  • Agent teams — a mesh. Multiple sessions as peers that message each other both ways. For genuine coordination. Experimental and expensive.
  • Git worktrees — parallel lanes. Separate working directories you develop in independently and merge at the end. For file isolation, not agent chatter.

Toggle the visualizer above through all three before reading on — the shapes are the fastest way to keep them straight.

7.2 · Subagents — isolation inside one session

Workers with their own context windows, spun up by the lead, reporting results back. The cheapest, most stable form of parallelism.

The key insight ties straight back to Module 03: a subagent has its own context window, so you can send it to do noisy, token-heavy work — reading 40 files to answer one question — and only the answer comes back to your main window. It’s context engineering as delegation.

  • One-way. Subagents report to the lead; they don’t talk to each other, and they cannot spawn their own subagents.
  • Configured in .claude/agents/ (or ~/.claude/agents/) as markdown + YAML: name, description (the routing signal), a tight tools allowlist, model per-subagent, permissionMode.
  • Built-ins: Explore (read-only, fast, cheap model — great for codebase search), Plan (research in plan mode), general-purpose.
  • Cost: roughly 4–7× a single session’s tokens. Instrument with the SubagentStart / SubagentStop hooks from Module 06.

7.3 · Agent teams — coordinating peers EXPERIMENTAL

Multiple sessions that talk to each other both ways. The most powerful and the least mature.

A team has a lead plus teammates with a shared mailbox for bidirectional peer-to-peer messaging. The lead orchestrates via primitives like TeamCreate, TaskCreate, SendMessage, TaskList.

enable (research preview)
$ export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
# requires Claude Code v2.1.32+ and Opus 4.6+
# teammates inherit the lead's permission mode
Cost & caveats — read before you reach for this

Agent teams run at roughly 15× a normal session’s token usage, and are a flagged research preview. Known limits: in-process teammates can’t be resumed (/resume won’t restore them), task status can lag and stall dependents, shutdown is slow, you can run one team at a time with no nesting, and open bugs have team tools leaking into standalone subagents. Use it when coordination is the actual requirement — not because “more agents” sounds faster.

7.4 · Git worktrees — filesystem isolation

Not an agent mechanism at all. A git feature that lets multiple sessions work without stepping on each other’s files.

parallel work, isolated checkouts
$ claude --worktree backend   # isolated checkout: .claude/worktrees/backend/
# branch: worktree-backend · shares git history, isolates working dir

# .worktreeinclude (gitignore syntax) copies .env etc into each new worktree
  • Each worktree is a separate working directory on its own branch — run a Claude session in each, no file collisions.
  • The desktop app auto-creates a worktree per session; you can also enter one mid-session.
  • No communication between them by design — you integrate by merging branches at the end (resolving conflicts as usual).
  • Cleanup: background/subagent worktrees are swept after cleanupPeriodDays if clean; --worktree ones are never auto-removed — git worktree remove them yourself.

7.5 · Choosing — the framework

Answer one question and the choice usually falls out.

  • Keeping heavy/noisy context out of your main window, within one task? → subagent.
  • Independent parallel tasks that touch files and merge later? → worktrees.
  • Workers that must coordinate with each other mid-flight? → agent teams (and accept experimental + ~15×).
  • Just want reusable instructions? → you don’t need parallelism at all — that’s a skill (Module 04).
They compose

The common production pattern is worktrees + subagents: a session per feature in its own worktree, each using subagents for exploration so its main window stays clean. Teams are the rare third layer, added only when the features genuinely can’t be developed in isolation.

7.6 · The hidden cost: you become the bottleneck

Parallelism multiplies tokens — and it multiplies the thing tokens can’t buy: your attention.

Three subagents or a five-Claude team produce three-to-five times the output you then have to review. The constraint stops being compute and becomes you, reading diffs. That’s why the honest default for most work is still one Claude with good context hygiene (Module 03) and solid hooks (Module 06). Reach for parallelism when the work is genuinely independent or genuinely needs isolation — never for its own sake.

Bridge to Module 08

“You-as-the-bottleneck” is one of the named scaling challenges of agentic engineering. The fix isn’t more agents — it’s better validation (so you trust the output without reading every line) and compounding your setup so each task gets cheaper. That’s the finale.
