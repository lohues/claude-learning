MODULE 06

Hooks

● core ~26 min read deterministic enforcement 6 lessons · 1 check

The persuade-vs-guarantee distinction, where hooks fire across the lifecycle, the anatomy (matcher + exit codes), how scope follows file location, the three hooks worth writing first, and when not to reach for one.

6.1 · Persuade vs guarantee

The single most important distinction in the whole course, and the reason this module exists.

A CLAUDE.md line, an auto-memory note, a skill description — all of these are context. Claude reads them and follows them most of the time. But “most of the time” is a probability, not a guarantee, and for a safety rule a single miss is a breach. A hook is deterministic code: it runs on an event, and if it exits with the blocking code, the action does not happen. Full stop. The gate above is the whole point — the CLAUDE.md side leaks a few times out of 25; the hook side never does.

The mental model

CLAUDE.md is a sign that says “please don’t.” A hook is a locked door. You want signs for guidance and doors for the handful of things that must never happen.

6.2 · Where hooks fire

Hooks attach to lifecycle events. There are around 27 of them — but you’ll use maybe six.

The headline events, in roughly the order they occur:

SessionStart
session opens; can inject context
UserPromptSubmit
you send a prompt; can add or block
PreToolUse
before any tool runs — the gate
PostToolUse
after a tool runs; can rewrite output
Notification
permission/idle prompts — desktop pings
Stop
Claude tries to finish; can force more
SessionEnd
session closes; cleanup

The star is PreToolUse — it fires before a tool executes, so it’s the only place you can stop an action before it happens. (Recall Module 02’s evaluation order: PreToolUse hooks run first, before even deny rules.)

+ The full event list (you don’t need to memorize this)

As of Claude Code 2.1.x there are ~27 distinct hook events — the larger counts you’ll see elsewhere include subtype matchers. Many recent ones exist for agent teams and worktrees (Module 07):

The number drifts upward release to release. Don’t learn the list; learn PreToolUse, PostToolUse, and Stop, and look up the rest when you need them.

6.3 · Anatomy: matcher + exit codes

A hook is configured in settings.json, filtered by a matcher, and communicates back through its exit code.

.claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",        # filter: only Bash tool calls
        "hooks": [
          { "type": "command", "command": ".claude/hooks/block-force-push.sh" }
        ]
      }
    ]
  }
}
block-force-push.sh
#!/usr/bin/env bash
input=$(cat)                      # hook receives JSON on stdin
if echo "$input" | grep -q -- '--force'; then
  echo "Force-push blocked by policy." >&2   # stderr → fed back to Claude
  exit 2                          # 2 = BLOCK. only 2 blocks.
fi
exit 0                            # 0 = allow
  • Types: command (most common), plus http, mcp_tool, prompt, agent.
  • matcher filters by tool name (Bash, Edit, Write) or session/notification subtype; an optional if field narrows further with permission-rule syntax.
  • Exit codes: 0 = success (stdout can become context on SessionStart / UserPromptSubmit); 2 = blocking error, stderr is fed back to Claude. PreToolUse can also emit JSON to allow/deny/ask and even rewrite the tool input; PostToolUse can rewrite output.
Footgun · only exit 2 blocks

An exit 1 is treated as a non-blocking error — Claude logs it and proceeds anyway. If your guard “detects” the bad command but it runs regardless, you almost certainly exited 1 instead of 2. This is the most common hook bug there is.

6.4 · Scope follows file location (and hooks merge)

Same hierarchy as permissions in Module 02 — but with one key difference in how conflicts resolve.

  • ~/.claude/settings.json — your hooks across all projects.
  • .claude/settings.json — project hooks, committed and shared with the team.
  • .claude/settings.local.json — personal, gitignored.
  • Managed/enterprise policy, plugin hooks/hooks.json, or skill/agent frontmatter (those last ones active only while that component runs).
Hooks merge — they don’t override

Unlike skills (where the most specific scope wins), all matching hooks fire, from every source. Your personal formatter and the team’s security guard both run. Good for layering defenses; worth remembering when you’re wondering why a hook ran “twice.” A subagent’s Stop hook auto-converts to SubagentStop so it fires at the right boundary.

6.5 · The three hooks to write first

You don’t need 27. You need these.

  • PreToolUse guard. Block the few things that must never happen — force-push to main, edits to .env, rm -rf outside a sandbox. The locked door. (Pairs with Module 02 deny rules but adds real logic.)
  • PostToolUse formatter / linter. Auto-run prettier/eslint/gofmt after every edit so style is never a review comment again. Mechanical correctness, enforced for free.
  • Stop “tests must pass.” A Stop hook that runs the suite and, on failure, blocks the stop — making verification non-optional. This is the closed loop Module 03 hinted at and Module 07 scales.
Also handy

Notification hooks for desktop pings when Claude needs you (use the terminalSequence field, since hooks have no controlling terminal). Inspect everything with /hooks; kill all of them in one shot with disableAllHooks: true when you need a clean run.

6.6 · When NOT to hook

Determinism cuts both ways. A hook runs every time — including when it’s wrong.

  • Don’t hook what doesn’t need a guarantee. Guidance and preferences belong in CLAUDE.md. Over-hooking adds latency to every action and makes sessions brittle.
  • A bad hook breaks every session. Hooks run with your shell access; a syntax error or a slow command degrades everything. Test in isolation before committing.
  • Hooks are a security surface. They execute commands with your privileges — quote variables, validate stdin, and never pipe untrusted content into one. A careless hook is its own vulnerability.
The rule of three, completed

Durable facts → CLAUDE.md. Repeatable procedures → skills. Rules that must hold every time → hooks. You’ve now got all three corners. Most messy Claude Code setups are just one of these doing another’s job — procedures crammed in CLAUDE.md, or a safety rule left as a polite note.

Tick as you go — these feed your meter.
