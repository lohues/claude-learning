One loop runs everything.

gatherReads the repo, the error, the files it needs — pulling just enough context to act.
actEdits files, runs commands, calls tools. The part people fixate on — and the least interesting.
verifyRuns the tests, re-reads the output, checks its own work — then loops back if it’s wrong.

Master verify and you’ve mastered Claude Code. Everything in this course is, ultimately, about making that step trustworthy enough to stop watching.

MODULE 01

Ground Control

● core ~25 min read install + first run 4 lessons · 1 check

What Claude Code actually is, the loop it runs, how to get it on your machine, and what your first real session looks like. The boring-but-load-bearing module.

1.1 · What it is, precisely

Not a chatbot with a code theme. A terminal-native agent that runs a real action loop against your real filesystem.

Claude Code lives in your terminal (and as a desktop/IDE app). You give it intent in plain language; it reads your codebase, plans, then uses actual tools — Edit, Write, Bash — to change files and run commands, checking results and iterating until the job is done. That cycle is the agentic loop: gather → act → verify.

The distinction that matters: a chat tool hands you text to paste; Claude Code does the thing and then tells you whether it worked. Which is exactly why the rest of this course is mostly about control — permissions, context, hooks — rather than prompting.

Anchor

Anthropic frames the loop as “gather context → take action → verify results,” and the day-to-day workflow as Explore → Plan → Code → Commit. Keep that four-beat in your head; it’s the rhythm of a good session.

1.2 · Getting it installed

One requirement, one preferred installer, and a deprecated path you should skip.

  • Node.js 18+ is the baseline runtime requirement.
  • Runs on macOS, Linux, and Windows. Some features (notably /chrome) want WSL on Windows — we’ll hit that in Module 05.
  • The native installer is now preferred; the old npm install -g route is deprecated. Use one of these:
install
# macOS / Linux — native installer (preferred)
$ curl -fsSL https://claude.ai/install.sh | bash

# macOS — Homebrew, also supported
$ brew install --cask claude-code

# then authenticate (Pro / Max / Team sub, or Console billing)
$ claude
✓ logged in — ready in ~/your-project
Gotcha

Auth needs an active plan: a Claude Pro / Max / Team subscription, or an Anthropic Console account with billing on. A free account won’t run it. The course’s own prerequisites say “Pro plan or higher” for this reason.

1.3 · Your first session

Launch in a repo, talk to it like a competent new hire, and watch the loop run.

Run claude inside a project directory. Then describe a goal — not step-by-step instructions. It explores, proposes, and (once you allow it) acts.

~/search-engine — claude
> the unicode test in search.ts is failing — find out why and fix it

● read search.ts (160 lines)
● read tokenizer.ts (240 lines)
● ran failing test → reproduced
root cause: regex /\w+/ doesn't match unicode
● patched tokenizer.ts:42  →  /\p{L}\p{N}+/gu
✓ tests: 47 passed (was 44)
✓ types: ok

Notice the shape: it gathered (read two files), acted (patched one line), and verified (re-ran the suite, reported the delta). You didn’t tell it to run the tests — verification is the default instinct you want to reinforce, never suppress.

+ Slash commands & the keys worth knowing on day one
  • /init — scans the repo and drafts a CLAUDE.md (Module 03).
  • /context — shows how full the context window is, by category (Module 03).
  • /clear — wipes the conversation to reclaim context.
  • /permissions — view and edit what it’s allowed to do (Module 02).
  • Shift+Tab — cycles permission modes live: default → acceptEdits → plan. The single most-used key combo. Full treatment in Module 02.
Gotcha · runaway loops

An agent that keeps acting is an agent that keeps spending. Cap it: --max-turns N limits loop iterations and --max-budget-usd X stops on cost. Use these from the start on anything unattended.

1.4 · The mental model to keep

Three habits that make every later module easier.

  • Goals, not procedures. Say what “done” looks like; let it find the path. You’ll intervene on direction, not keystrokes.
  • Demand evidence. “Show me the passing test / the screenshot,” not “did it work?” This one instinct is the seed of Module 06’s self-correcting loop.
  • Watch the context, not just the output. The loop degrades as its working memory fills. That’s Module 03, and it’s the highest-leverage thing in the whole course.
