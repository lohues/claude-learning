MODULE 03

Context Engineering

● core highest leverage ~28 min read 6 lessons · 1 check

What the context window actually is, why it rots as it fills, the three commands that manage it, and the three places durable knowledge belongs — CLAUDE.md, auto memory, and (a preview of) hooks.

3.1 · Finite working memory & context rot

The window is large — 200K tokens on current models — but it’s working memory, not a hard drive.

Because every token attends to every other token, the model’s grip on any single detail weakens as the window fills. Anthropic calls this context rot. The symptom you’ll feel in practice: Claude “forgets” an instruction from 40 turns ago, or quietly reintroduces a bug you already fixed. That’s not the model getting dumber — it’s the window full of noise drowning the signal.

So the goal of this whole module inverts most people’s instinct: not “give it everything just in case,” but the smallest set of high-signal tokens that still gets the job done.

Anchor

Anthropic frames context engineering as finding “the smallest possible set of high-signal tokens that maximize the likelihood of the desired outcome.” Hold that sentence; the rest of the module is mechanics for living it.

3.2 · Read the gauge first: /context

You can’t manage what you can’t see. /context breaks down what’s eating the window, by category.

/context
> /context
context window — 200,000 tokens
  system + tools        14,200   ▓▓
  MCP tool definitions  26,600   ▓▓▓▓  ← bigger than you think
  CLAUDE.md              3,100   ▓
  files read this session 41,000 ▓▓▓▓▓▓
  conversation          22,400   ▓▓▓
total in use: 107,300 (54%)

The reliable surprise: it’s rarely the conversation that fills the window — it’s MCP tool definitions and large tool/file outputs. A single connected server can quietly cost tens of thousands of tokens before you type a word (the full reckoning is Module 05). Run /context early and whenever things feel sluggish.

3.3 · The two reclaim commands

When the gauge climbs, you have two levers — and they do different things.

  • /clear — wipes the conversation history entirely. The clean reset between unrelated tasks. Cheap, total, lossy.
  • /compact — summarizes the session so far into a compact form, freeing space while keeping the gist. The mid-task lever when you’re deep but the window’s filling.
Survives compaction

Your root CLAUDE.md is re-read from disk and re-injected after a compact, so project memory persists. Nested CLAUDE.md files do not automatically come back — another reason to keep durable rules in the root file.

Gotcha · don’t wait for auto-compact

Claude Code auto-compacts around 95% capacity, dropping old tool outputs first. By then you’ve already been operating in the rot zone for a while. Compact deliberately at a natural breakpoint (~60–70%), not when the system forces it mid-thought. And remember /compact is lossy — don’t do it in the middle of subtle surgery.

3.4 · CLAUDE.md — durable project memory

The file that loads every session. Treat it like a constitution, not a notebook.

Run /init and Claude scans the repo and drafts one. The discipline is knowing what earns a place in it — because every line you add is a line loaded into every future window.

Belongs in CLAUDE.md

  • Build / test / lint commands
  • Architecture in two lines
  • Naming & style conventions
  • The non-obvious “golden rules”
  • Key paths & entry points

Does NOT belong

  • Step-by-step procedures → skills (Mod 04)
  • Anything readable from the code itself
  • Per-task or one-off details
  • Hard rules that must never break → hooks (Mod 06)
  • Walls of text “just in case”

Nesting & imports: a CLAUDE.md in a subdirectory loads on demand when Claude works there, and @path/to/file imports pull in other files up to 5 hops deep. Use this to keep the root lean and push area-specific detail down to where it’s relevant.

The 200-line rule

Past roughly 200 lines, a CLAUDE.md both eats noticeable context and gets followed less reliably. A bloated memory file is paradoxically a less-obeyed one — the same rot, applied to your own instructions. Keep it short and high-signal. If it’s growing, that’s a signal something belongs in a skill or a hook instead.

3.5 · Auto memory — the “Second Brain”

Memory Claude writes for itself, persisting across sessions.

Beyond the CLAUDE.md you author, Claude Code can maintain its own markdown memory store — browsable and editable with /memory — where it records learnings so they survive between sessions (the course’s “Second Brain” / “Auto Memory”). The same ~200-line discipline applies; it’s context too.

The load-bearing caveat

Both CLAUDE.md and auto memory are context, not enforced policy. Claude reads them and usually follows them — but “usually” isn’t “always.” For a rule that must hold 100% of the time (never read .env, always run tests before commit), a memory line is the wrong tool. You want a hook — deterministic enforcement, covered in Module 06. This is the single most important distinction in the course: memory persuades; hooks guarantee.

3.6 · The discipline, in three habits

Everything above collapses into this.

  • Smallest high-signal set. Don’t paste the whole spec; give Claude identifiers and let it retrieve just-in-time. Lightweight pointers beat pre-loaded dumps.
  • Compact at breakpoints, not at 95%. Treat context like a workspace you tidy between tasks — proactively, on your terms.
  • Right home for each thing. Durable facts → CLAUDE.md. Repeatable procedures → skills. Rules that must never break → hooks. Put it where it belongs and the window stays clean.
+ Why “just-in-time retrieval” beats front-loading

Front-loading everything assumes you know in advance exactly what Claude will need — you usually don’t, and the surplus becomes rot. Just-in-time means Claude holds lightweight identifiers (file paths, function names, a table of contents) and pulls the full content only at the moment it acts. It’s the same instinct as a senior engineer who doesn’t memorize the codebase but knows exactly where to look. Anthropic’s context-engineering guidance leans hard on this pattern, alongside compaction and structured note-taking, as the way agents stay coherent over long horizons.
