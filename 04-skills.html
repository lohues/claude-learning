MODULE 04

Skills

● core ~26 min read SKILL.md anatomy 6 lessons · 1 check

What a skill is, why progressive disclosure makes them nearly free, how a SKILL.md is built, the one part everyone botches (the description), where they live, and how they compose.

4.1 · What a skill is

A folder with a SKILL.md inside it. That’s the whole primitive.

A skill packages a repeatable workflow — “how we cut a release,” “how to merge PDFs,” “how to triage a log” — as instructions Claude pulls in when the task calls for it. It’s the direct answer to the question Module 03 left open: procedures don’t go in CLAUDE.md (that bloats every session), and they don’t stay in your head (Claude can’t see them). They go in a skill, where they sit dormant until needed.

Lineage

Skills are the modern replacement for the legacy .claude/commands/ mechanism. If you’ve seen old custom-command setups, skills are the same idea grown up — with triggering, bundled files, and composition.

4.2 · Progressive disclosure — why they’re nearly free

The mechanism the ladder above demonstrates. Three tiers, each loaded later and only if needed.

  • Tier 1 — metadata. The skill’s name + description only. Always pre-loaded into the system prompt so Claude knows the skill exists. Tiny — tens of tokens.
  • Tier 2 — body. The full SKILL.md instructions. Loaded only when the skill triggers. A few hundred tokens, paid once, only when relevant.
  • Tier 3 — bundled files. scripts/, references/, assets/. Read on demand when the body reaches for them; scripts execute outside the window entirely. Effectively unbounded supporting context at near-zero cost until used.
The payoff

This is why you can install dozens of skills and barely move idle context — you only ever pay metadata for the ones you don’t use. It’s Module 03’s “smallest high-signal set” principle, built into the primitive. Skills are how you give Claude enormous capability without filling its window.

4.3 · Anatomy of a SKILL.md

YAML frontmatter on top, markdown instructions below, optional folders alongside.

.claude/skills/pdf/
pdf/
├── SKILL.md        # required: frontmatter + body
├── scripts/        # optional: executable helpers
│   └── merge.py
├── references/     # optional: docs loaded on demand
│   └── forms.md
└── assets/         # optional: templates, files
SKILL.md
---
name: pdf
description: Merge, split, and fill PDF files. Use whenever
  the user mentions a .pdf, asks to combine/merge PDFs,
  or fill a PDF form.
---

# Working with PDFs
1. Validate each input is a real PDF.
2. Merge in the requested order using pypdf.
3. For forms, see references/forms.md.
4. Run scripts/merge.py and verify the page count.

name and description are the only required fields. Everything else — the body, the folders — is optional scaffolding. Keep the body lean: roughly under 500 lines (~1,500–2,000 words). Past that, push detail into references/ where it loads on demand.

4.4 · The description is the trigger

The single most important — and most botched — line in any skill. It’s the only part Claude sees before deciding whether to load the rest.

Claude matches the user’s request against descriptions (Tier 1). A vague description never fires; the skill sits there, perfectly written, never used. State what it does AND when to use it, with concrete trigger phrases.

Won’t trigger

description: Helps with PDF tasks.

Too vague. What tasks? Triggered by what words? Claude under-fires on this.

Fires reliably

description: Merge, split, and fill PDF files. Use whenever the user mentions a .pdf, asks to combine/merge PDFs, or fill a form.

Concrete verbs + explicit trigger conditions.

Be a little pushy

Anthropic’s own skill-creator guidance notes Claude tends to under-trigger, so descriptions should lean assertive about when to activate. Other authoring habits that help: name skills as gerunds (“merging-pdfs”), and in the body explain the why behind a rule rather than barking bare ALL-CAPS imperatives — reasons generalize to cases you didn’t foresee; commands don’t.

4.5 · Where they live & who can invoke them

Three locations by scope, plus two frontmatter switches that control invocation.

  • ~/.claude/skills/ — user scope. Available in every project. Your personal toolkit.
  • .claude/skills/ — project scope. Committed to the repo, shared with the team.
  • Nested project dirs — loaded on demand when Claude works in that area.

On a name collision, the more specific scope wins (project overrides user). Two switches worth knowing:

  • disable-model-invocation: true — Claude won’t auto-trigger it; you invoke it yourself as a slash command (e.g. /deploy). For things that should only run when you say so.
  • user-invocable: false — the inverse: background knowledge Claude can pull in, but you can’t call directly.
Gotcha · reload behavior

Editing a SKILL.md’s text is picked up live in-session. But adding a new top-level skill directory typically needs a restart before Claude sees it. If your fresh skill isn’t firing, restart before you debug the description.

4.6 · Composability & built-ins

Small skills that call other skills beat one monolith.

Skills compose — one can invoke another — so the right unit is small and single-purpose, not a 600-line do-everything skill. A release skill can lean on a changelog skill and a test-runner skill. Claude Code also ships built-in skills for common document and workflow tasks, so check what already exists before authoring. This composition instinct is the seed of compound engineering — the Module 08 idea that each thing you build makes the next one cheaper.

+ Skill vs. CLAUDE.md vs. hook — the three-way decision

CLAUDE.md = durable facts, always loaded (conventions, build commands). Skill = a procedure, loaded when triggered (how to do a multi-step task). Hook = a rule that must hold every time, enforced deterministically (Module 06). The mistake is reaching for CLAUDE.md for all three. A bloated CLAUDE.md full of step-by-step procedures is the #1 symptom of not knowing this split — move the procedures to skills and the file gets short and well-followed again.

Try it yourself

Tick as you go — these feed your meter.

  • Create .claude/skills/<name>/SKILL.md with name + descriptionstart with one repeatable thing you do weekly
  • Write a description with explicit trigger phrases, then test it firessay one of the trigger phrases and watch it load
  • Move a multi-step procedure out of CLAUDE.md into a skillwatch CLAUDE.md get shorter
  • Add a references/ file and reference it from the bodyconfirm it only loads when that path is hit
  • Make one skill disable-model-invocation and run it as a slash commande.g. a /deploy you control
